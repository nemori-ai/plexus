# Plexus integrations — let mainstream coding agents USE Plexus

This directory is the **integration layer**: how mainstream coding agents (Claude
Code and OpenAI Codex CLI) actually USE the capabilities a local Plexus gateway
exposes. The two agents integrate two different ways:

- **Claude Code** — the gateway **compiles a plugin per-agent**. An admin connects
  an agent (console wizard or `POST /admin/api/agents/connect`); the gateway renders
  that agent's granted cap-set + its live Floor into a ready-to-install plugin and
  serves a **one-command install**. The plugin ships a per-agent launcher
  `plexus-<agentId>` and a compiled skill. See
  [`docs/tutorials/connect-an-agent.md`](../docs/tutorials/connect-an-agent.md).
- **Codex** — the portable path: an **AGENTS.md instruction block + a shared `plexus`
  command on PATH**, driven by `codex exec`. Codex enrolls once (one-time code → PAT)
  and then lists/invokes. See [`codex/`](./codex/).

Plexus is a local capability gateway (`bin/plexus`, loopback `127.0.0.1:7077`) that
speaks its own AI-native protocol (`DISCOVER → handshake → grant → invoke`) plus a
**usage-skill layer MCP does not have**. But an agent never speaks that wire by hand:
its `plexus` command (compiled launcher for CC, shared CLI for Codex) is its
**complete and only** interface — enroll, discover, and invoke are all subcommands.

## Two credentials — keep them straight

- **Connection-key** (`plx_live_…`) — the **admin/management** credential. It gates
  the console and `/admin/api/*`. **Agents never see it.**
- **Per-agent PAT** — the **agent's** durable credential, redeemed **once** from a
  one-time enrollment code (`plx_enroll_…`) that rides the install / is handed to the
  agent to `enroll`. The PAT-auth handshake binds the real `agentId`.

An agent authenticates with its own PAT and only ever with its own PAT. It never
reads a connection-key, never hand-rolls an HTTP request, and never guesses a token —
the command handles auth internally and hides it. (Full model:
[`docs/design/security-model.md`](../docs/design/security-model.md).)

## The core integration insight: Plexus is NOT an MCP server

The obvious-looking path — drop a `{ "mcpServers": { ... "url": ".../mcp" } }`
entry into Claude Code's `.mcp.json` or Codex's `config.toml` — **does not work**,
because Plexus does not expose an `/mcp` Streamable-HTTP wire. MCP is an
*ingestion transport* Plexus runs as a **client** (to pull tools from MCP servers
into its own entry model); Plexus's *outward* surface is its own
discover→grant→invoke protocol, reached through the `plexus` command.

The per-agent Claude Code plugin is **not** checked in here — it is generated by the
gateway's renderer (`packages/runtime/src/integration/render-plugin.ts`) from the
committed engine (`tools/plexus-cli/plexus`) and the [P] skill body template. A
build-time verifier gates every rendered plugin against the Floor (Inv VI). So there
is deliberately **no** hand-written `skills/use-plexus/SKILL.md` in this tree.

## Claude Code — compiled per-agent by the gateway

**Mechanism.** A Claude Code *plugin* is a directory with `.claude-plugin/plugin.json`
plus, at the plugin root, any of: `skills/<name>/SKILL.md`, `commands/`, `hooks/`,
`.mcp.json`, and a `bin/` dir whose executables join the Bash tool's PATH while the
plugin is active. The gateway uses exactly those mechanisms — a **SKILL** (teach) +
a **`bin/` launcher** (run) — but the artifact is **rendered per agent**, not written
by hand.

**The flow (from the user's POV):**

1. **Connect the agent** (admin, holds the connection-key) — the console's
   *Connect an agent* wizard, or `POST /admin/api/agents/connect`. This mints a
   one-time enrollment code (`plx_enroll_…`) and grants a starting cap-set to the
   agent as **standing** grants (the human approval, done once).
2. **One-command install** — the console shows the copy-able install from
   `GET /integration/:agentId` (public bootstrap at `.../install.sh`). It installs a
   plugin whose files are compiled for this one agent.
3. **The agent uses it** — the plugin puts a **`plexus-<agentId>` launcher** on the
   Bash PATH (its own bundled, version-pinned engine — never a bare/global `plexus`,
   which could be shadowed on PATH and authenticate as the wrong agent). Its
   subcommands are the agent's entire surface:
   - `plexus-<agentId> enroll <code>` — redeem the one-time code for this agent's PAT
     (once; the install does this for you via the code that rode the install command).
   - `plexus-<agentId> list` — what is callable now (standing grants) vs what needs
     the owner's approval, including capabilities exposed after compile time.
   - `plexus-<agentId> <capabilityId> [args]` — invoke (positional / `key=value` /
     `--input '<json>'`; `--json` to parse; `--purpose` to state *why* when a call
     may pend).

**The compiled skill is a projection over the always-present self-describing floor.**
The auth/invoke core (enroll → PAT → handshake → token → invoke) lives entirely
inside the bundled engine and is templated, never LLM-authored; the skill body is
pedagogy only. RULE the skill enforces: the launcher is the agent's complete and only
interface — never hand-roll HTTP, never guess auth.

**Acceptance:** connect an agent granting `obsidian.vault.read`, run the shown
one-command install, then in a CC session ask "read my Obsidian note
`Projects/Plexus.md` via Plexus" — the compiled skill runs
`plexus-<agentId> obsidian.vault.read path=Projects/Plexus.md` and returns the real note.

## Codex — AGENTS.md + the shared `plexus` command

**Mechanism (codex-cli 0.141.0).** Codex reads instruction context from **AGENTS.md**
(global `~/.codex/AGENTS.md` + per-project, walking git-root→cwd; combined cap
`project_doc_max_bytes`, default 32 KiB). It has a capable shell. Codex is **not** a
compiled-plugin agent — the right shape is AGENTS.md (teach) + a shared `plexus`
command on PATH (run), driven non-interactively by
`codex exec [--sandbox …] "<prompt>"`.

**Why AGENTS.md + shell (not an MCP entry).** Same insight: Plexus has no `/mcp`
wire, so a `[mcp_servers.plexus]` entry has nothing to connect to.

**What shipped (`integrations/codex/`):**

```
integrations/codex/
├── AGENTS.plexus.md     # the drop-in instruction block: enroll → list → invoke,
│                        #   the "command is your interface / you never handle the
│                        #   credential / enroll if unenrolled" rule. Marker-guarded.
├── bin/plexus           # the launcher Codex puts on PATH (shim → shared CLI engine)
├── setup.sh             # idempotent: symlink bin/plexus onto PATH + append the block
└── setup.md             # the full setup walkthrough
```

The Codex agent authenticates with its **own per-agent PAT**, redeemed once from a
one-time enrollment code — it never reads the connection-key. Connect the agent
(pick **Generic / other agent** in the console, or `POST /admin/api/agents/connect`)
to get the code, then `plexus enroll <code>` once. Thereafter the AGENTS.md block
teaches the same discipline the compiled CC skill does: **the command is your
interface; you never handle the credential; enroll if unenrolled**; use `plexus list`
to discover, `plexus <capabilityId> [args]` to invoke.

**Acceptance:** with the AGENTS.md block installed, `plexus` on PATH, and the agent
enrolled, `codex exec "use Plexus to read my note Projects/Plexus.md"` drives
`plexus list` → `plexus obsidian.vault.read …` and returns the real note. A
deterministic, no-LLM CI smoke (`tests/integrations-codex-e2e.test.ts`) drives the
same shim by bare name `plexus` and asserts the closed `ErrorCode` union.

## Confirmed agent-mechanism findings

- **Claude Code** consumes external capability via **plugins** (`plugin.json` +
  `skills/SKILL.md` + `bin/` on Bash PATH + optional `.mcp.json`). ⇒ the gateway
  **compiles** such a plugin per-agent: a rendered SKILL (teach) + a `plexus-<agentId>`
  launcher (run), delivered as a one-command install. No `/mcp` entry.
- **Codex CLI** consumes external capability via **AGENTS.md** instruction context
  (global + project) and a capable shell; non-interactive `codex exec`. ⇒ `codex/` =
  AGENTS.md (teach) + `bin/plexus` on PATH (run). No `/mcp` entry.
- **Both** reach the same gateway protocol through a `plexus` command over the shell —
  no `/mcp` wire, no per-agent protocol re-implementation, and the agent authenticates
  with its own PAT (never the connection-key).
