# Plexus — Design Context

## Design Context

### Users
Developers and power-users running the **Plexus local capability gateway** on their own
machine. They open this panel to govern exactly which local capabilities an AI agent may
discover and call: what is exposed vs. hidden, read-only vs. read-write, what tokens are
live, and what has actually happened (the audit trail). Context of use: at their own desk,
terminal-adjacent, often alongside an agent session. This is a **local developer admin
tool / control surface**, not a marketing site and not a multi-tenant SaaS dashboard.

The dominant job: make a **trust decision** legible and reversible — "what am I about to
let this agent touch, with what verbs, and can I take it back."

### Brand Personality
Three words: **custodial, evidentiary, mechanical.** Calm authority. The panel behaves
like a *permission ledger* — a register you sign and can revoke. Confident, precise, never
flashy. Security framing must always read: default-deny, default-read-only, per-capability
grants, fully revocable, fully audited.

### Aesthetic Direction
- **Theme: dark.** Warm-neutral base tinted toward amber ("ledger / lamplight"), built in
  OKLCH. The audience operates this at a desk in a developer workflow; dark suits the
  observability/control-surface context. NOT the banned cyan-on-dark neon AI look.
- **Mental model: a custodial ledger**, not a card grid. Capabilities are rows in a
  register with an exposure rail and verb stamps. The audit trail reads like an
  evidentiary log.
- **Type:** Hanken Grotesk for UI/body (humanist, slightly mechanical warmth); Spline Sans
  Mono ONLY for machine identifiers (capability ids, jti, scopes) — mono is meaningful, not
  decoration.
- **Color is the trust signal (the 10% accent):** amber = active/brand, desaturated green
  = granted/allow/exposed, clay-red = revoke/deny, slate = hidden/neutral.
- **Anti-references:** generic bootstrap admin, AI-template dashboards, hero-metric tiles,
  colored side-stripe cards, gradient text, glassmorphism, neon glow.

### Design Principles
1. **Trust state is the loudest signal.** Exposed/hidden, read/write, granted/revoked must
   be readable at a glance through color + position, never buried in body text.
2. **The ledger, not the grid.** Capabilities are register rows; the audit is an evidentiary
   log. Avoid repeated identical cards.
3. **Mono means machine.** Monospace is reserved for ids/jti/scopes the user copies or
   correlates — never as a generic "techy" texture.
4. **Default-deny is visible.** Nothing is exposed until the user chooses; the issue-grant
   action always tallies precisely what is about to be authorized.
5. **Reversibility is first-class.** Revoke is always one confident action away, and its
   outcome is reflected immediately.
