## 一、结论先行：这不是一次重写

**这是「形式化 + 扩展 + 换肤」，不是重写。** 这是整份方案里最让人安心的判断，先讲清楚：

我们刚刚做完的那套鉴权/授权逻辑——授权（grant）、约束（constraint）、任务包（bundle）、审计（audit）、来源（source）——**原封不动保留**，一行都不用动。它们全部活在内存里的 `GatewayState`（`src/core/state.ts:38`）这一束已接好的状态里，只能通过 HTTP/SSE 接口触达，**完全不耦合任何 UI**。

之所以能做到「不重写」，是因为运行时**今天就已经是一个无头（headless）的回环（loopback）服务**：

1. **核心就是一个请求处理函数。** 「网关」是由 `createAppWithState(config)`（`src/core/server.ts:38`）构造的一个纯 Hono 应用。构造与监听彻底解耦：`server.ts` 只负责搭好 `Hono` 应用 + `GatewayState`，真正的端口绑定只发生在两个入口里（`src/index.ts:20`、`bin/plexus:256`），都通过 `Bun.serve({ fetch: app.fetch, … })`。一句话概括可分离性：**运行时是一个 `Request→Response` 的请求处理器，其余一切都是它的客户端。**

2. **Bun 专有代码几乎为零。** 对 `src/` + `bin/` 全量 `grep "Bun\."`，命中恰好只有 `Bun.serve`（3 处，其中 2 处是真入口，1 处是 `security.ts:33` 里的文档注释）。其余全是 Node 兼容 API（`node:child_process`、`node:fs`、`node:net`、`node:crypto`）。**唯一 Bun 专有的就是那一个监听调用，仅此 2 处。** 运行时事实上已经能在 Bun 与 Node 之间跨运行。

3. **已经有两类非内置客户端在跑，本身就是可分离性的活证据。** `plexus` 命令行（`integrations/cli/plexus-cli.ts`）和 Web 管理端（`management-client/`，React+Vite SPA）都已经是这套 HTTP API 的客户端。命令行从 `~/.plexus/connection-key` 读取连接密钥（无需粘贴），始终发回环 `Host` 头——这正是**任何**服务端客户端（含未来的 TUI）的模板。

4. **唯一与 Web 耦合的地方，薄、且只有一处。** 即 `admin.ts:648-673` 里的静态 SPA 托管（读 `management-client/dist`）。它是**附加的**——管理 API 在没有它时也能无头工作（缺失时优雅降级为 `NOT_BUILT_HTML`）。而且 tfix1 已经把安全护栏放宽为「任意回环主机、任意端口」，所以一个独立的回环客户端来源（比如 Electron 渲染进程跑在 `http://127.0.0.1:PORT`）现在就已经能通过 Host/Origin 校验——这一点是桌面模型的承重点，且**已经就位**。

**所以这次工作的实质是五件事：**
(a) 把运行时切成独立包 + 一个稳定的监听入口；
(b) 把现有两个 API 平面 + 一个推送通道，形式化为带版本号的**本地运行时契约**；
(c) 补上**唯一真正的缺口**——管理面的实时推送通道；
(d) 把 Web SPA 抽成独立客户端包，并新增一个 Electron 桌面客户端；
(e) 补完 Windows/Linux 的平台接缝。

鉴权、授权、流水线、协议线缆——**都不需要、也不建议重写。**

## 二、架构设计

### 2.1 核心运行时拆包

**核心运行时** = 今天的 `src/**`，无头、跨平台。把它切进 `packages/runtime`，配一个**受监督的单一 `plexus serve` 入口**。这个入口要做三件事：

- **机读就绪行（machine-readable ready line）**：进程启动后向 stdout 打一行可解析的就绪信息，如
  `PLEXUS_READY {"port":54321,"pid":…,"lraVersion":"1.0"}`，供监督方解析出真实绑定端口（临时端口时这点是关键）。
- **端口文件 `runtime.json`**：把选中的端口写到 `~/.plexus/runtime.json`（`{port, pid, lraVersion}`），让命令行/智能体在非默认端口时也能无需环境变量地发现它。
- **监听适配器接缝（listen-adapter seam）**：把那一个 `Bun.serve` 调用收拢进一个模块。这样万一未来某个客户端必须在进程内运行运行时（例如一个没有 Bun 二进制的纯 Node TUI 环境），同一个 `app.fetch` 就能交给 `@hono/node-server` 来服务，核心零改动。这是**便宜的保险**，建议在 P0/P1 阶段就做。

今天那两处 `Bun.serve` 调用点（`index.ts`、`bin/plexus`）就此收敛为一个受监督入口，同时保留 `bin/plexus` 作为人类启动器。

### 2.2 本地运行时契约 LRA v1（Local Runtime API）

定义 **LRA v1**——每个客户端（Electron / TUI / Web / 命令行）都讲的、稳定的、带版本号的本地契约。它就是**今天两个 API 平面 + 一个推送通道**的并集，冻结并版本化，在回环上以 HTTP + SSE 提供（WebSocket 可选）。纯 JSON、语言无关，所以将来一个 Rust/Go 的 TUI 或一个 Swift 客户端也能讲它。

**分层（复用，不重造）：**

- **智能体平面 = 原样不动。** 协议线缆（握手/授权/调用/事件/清单/扩展）保持协议 0.1.2 不变，只增不改。LRA 通过引用把它纳入；本身也是智能体的客户端直接用它。
- **管理平面 = 今天的 `/admin/api/*`，升版进 `/v1`。** 形状已经存在且在 `src/protocol/types.ts` 里有类型；把它们提升到共享的 `protocol` 包，让所有客户端都 import，于是契约**由编译器强制**。`/admin/api/*` 保留为 v1 别名，让现有命令行与 Web 管理端在过渡期照常工作。
- **推送平面 = 新增的管理事件流（这是唯一真正的缺口）。** 见下。

#### 唯一真正的「缺口」：管理面推送流

今天只有**智能体**有推送流（`GET /events` SSE）。**管理面没有推送**：Web 管理端和任何托盘要想知道「来了一个新审批」，只能去**轮询** `/admin/api/pending`。对一个要做托盘角标 + 原生通知的桌面应用来说，**轮询是错的**——这就是为什么推送流是这次唯一需要新造的运行时能力。

新增 **`GET /v1/events`——一条管理面 SSE 流**（智能体 `/events` 的镜像，受众过滤不同）。它在现有 `EventBus`（`events.ts`，本就是一个扇出）基础上，再发以下事件：

- **`pending_added` / `pending_resolved`【新事件类型】**：当某智能体的 `PUT /grants` 或 `POST /extensions` 产生一条待办项、以及该项被批准/拒绝时发出。这正是驱动**托盘角标 + 原生通知**（「智能体 X 想写入你的库……」，即 Mode-1）的事件。待办的叙述文本（`PendingNarration`, `types.ts:300`）已经存在，事件只需携带/引用它。
- **`audit_appended`【新事件类型】**：每条 `audit.write` 的投影，让仪表盘的「审计脉搏」无需轮询即可实时跳动。
- **重广播** `manifest_changed`、`token_revoked`、`source_status`（这些已在发布）。

实现成本很小：给 `PlexusEvent` 加上新事件变体，在待办项创建/解决与审计写入处发布它们，再加第二个订阅管理受众的 SSE 处理器即可。

**传输选 SSE，不选 WebSocket。** 服务端→客户端的单向流，正好就是托盘/通知/审计脉搏的需求；智能体 `/events` 平面已经证明 SSE 能轻松穿过回环护栏、无需升级握手。WebSocket 留作可选的未来（今天没有客户端需要双向流）。

### 2.3 嵌入模型：Bun 单文件可执行作为 Electron 的受监督边车

**Electron 主进程把 Plexus 运行时作为一个受监督的边车（sidecar）子进程拉起并监督。**

- **分离是真实的。** 运行时就是服务器管理员无头运行的那个同一份产物；桌面应用只是又一个客户端，只不过它同时拥有进程生命周期。
- **运行时原封不动。** 不往 Node/Electron-主进程里移植，不重写 `Bun.serve` 入口，不给已经工作的鉴权逻辑带来任何风险。
- **崩溃隔离干净。** 运行时崩了不会拖垮 UI；监督方重启它，渲染进程重连到同一份 `~/.plexus` 状态。

**为什么不把运行时移植进 Node/Electron-主进程？** Bun 耦合很小（仅 `Bun.serve`），移植技术上很便宜。但移植仍是**错的**：它会溶解掉所有者明确想要的分离（运行时将活在 Electron 主进程里，不再是服务器跑的那同一份产物）；它会把运行时的发布节奏绑死在 Electron 上；它会让每个 `node:*` 调用都要在 Electron 的 Node 下重新验证，并丢掉 Bun 的启动速度与单文件可执行打包能力。**但**正因为耦合极小，我们仍保留那个监听适配器接缝作为便宜保险。

**进程生命周期：** 启动时主进程拉起边车二进制，传入 `PLEXUS_PORT`、`PLEXUS_HOME`（默认 `~/.plexus`）、实例名。就绪靠「解析就绪行 + 轮询 `GET /v1/health` 直到 200」双保险。健康检查按间隔轮询 `/v1/health`，连续 N 次失败即判定死亡，指数退避重启并在托盘显示「运行时重启中…」。退出时发 SIGTERM、等待、最后 SIGKILL，并在 `will-quit` 处追踪 pid 杀掉防孤儿。用 Electron `requestSingleInstanceLock()` 保证第二次启动聚焦已有窗口而非再拉一个运行时。

**端口选择：** 默认保留 `7077`（与命令行、Claude-Code 插件的默认值连续），冲突则绑临时端口（`port: 0`）并从就绪行学到真实端口，写进 `runtime.json`。Unix 域套接字（UDS）作为未来的加固选项记录在案，v1 仍用 TCP 回环。

**免粘贴的受信客户端密钥交接：** 桌面应用是**那个**受信管理客户端，必须在用户不粘贴密钥的情况下向运行时认证。**推荐 v1 路径：Electron 主进程直接读 `~/.plexus/connection-key`（它有用户家目录的 fs 访问权，同一信任域），通过 `contextBridge` 预加载脚本注入给渲染进程；渲染进程从不读文件、从不显示密钥，只把它作为 `X-Plexus-Connection-Key` 发出。** 这正是 Web 管理端今天做的事，只是省掉了网络往返——零新增加密、复用已被验证的管理密钥护栏。渲染进程隔离：`contextIsolation: true`、`nodeIntegration: false`，密钥不进页面可达的全局变量。轮换走 `POST /v1/connection-key/rotate`，触发即丢弃全部站立会话/令牌（已接好，`state.ts:127`）。身份姿态不变（连接密钥才是本地单用户的信任边界，`agentId` 只用于划界/审计，不是加密认证）。

### 2.4 客户端矩阵

- **`desktop`（Electron）** —— 主推的规范体验。主进程：监督 + 托盘 + 原生通知；渲染进程：管理界面 + 仪表盘 + 引导。
- **`web-admin`（同一份 React SPA，降级保留）** —— 由运行时同源托管，作为可选的无头回退。对**无头 Linux 服务器**（在 TUI 落地前）和远程访问是近乎免费的正确答案。托盘/通知在 Web 构建里优雅降级为应用内的「审批」列表。不为 Web-only 抛光投入。
- **`cli`（已存在）** —— 今天的 `integrations/cli`，是 LRA 客户端的模板。
- **`tui`（未来，服务端）** —— 讲同一套 LRA + 事件流。它就是**又一个 LRA 客户端**：连到运营者直接启动的无头运行时（systemd 服务 / 裸 `plexus serve`，无 Electron），订阅 `GET /v1/events` 渲染审批收件箱、授权账本、审计尾巴。它认证方式与命令行完全一致。**TUI 不需要任何超出 LRA + 管理事件流的新运行时面。**

共享 `protocol` 包 = 由编译器强制的契约，运行时与每个客户端都 import 它。

### 2.5 跨平台 = 补两个隔离好的平台接缝桩（先 Linux）

`PlatformServices`（`protocol/types.ts:1729`）是单一的 OS 抽象接口：`resolveBinary`、`getEnrichedPath`、`locateLocalService`、`spawnProcess`、`resolveSecret`。选择器（`platform/index.ts:20`）按 `process.platform` 分派；`darwin.ts` 是实现，`win32.ts`/`linux.ts` 是带类型的抛错桩。没有任何 `process.platform` 判断泄漏出这个接缝。**所以跨平台 = 填两个桩，不是重新架构。**

每个实现要补：`getEnrichedPath`（捕获用户真实交互 PATH，Linux 用登录 shell 探测，Windows 读环境/注册表）、`resolveBinary`（`which` 等价，Windows 要尊重 `PATHEXT`）、`spawnProcess`（NDJSON 行帧器已 OS 中立，Linux 几乎照搬，Windows 要处理 `.cmd` shim 与引号）、`locateLocalService`（TCP 探测已跨平台）、`resolveSecret`（`~/.plexus/secrets/` 文件存储已跨平台）。一个 Windows 注意点：secrets 的 `chmod 0600` 在 Windows 是空操作，用平台接缝挡掉或接受空操作 + 后续 ACL 加固。

**先 Linux（服务器 + TUI 目标、更简单的接缝），再 Windows（桌面目标、spawn/PATHEXT 更多细节）。** 因为接缝是隔离的，填桩**不阻塞** Electron 工作，二者可并行。macOS 桌面可以先于 Windows 接缝就绪发布；Windows 桌面发布以 Win32 接缝就绪为闸门。

### 2.6 分发打包（一个 app，用户感知不到分离）

- **运行时打成单文件 Bun 可执行，作为 Electron 边车随包发。** `bun build --compile --target=bun-darwin-arm64|...|bun-windows-x64|bun-linux-x64` 产出每平台一个原生单文件可执行，用户机器**无需安装 Bun**（运行时自带 Bun）。electron-builder 把每平台运行时二进制作为 `extraResource` 随包发，主进程在开发/生产用同一套监督代码解析路径并拉起。
- **Web SPA 直接进 Electron 渲染进程**（`loadFile`/`loadURL` 到打包构建），桌面应用里**无需**由运行时托管（运行时的静态托管保留，只为可选的无头 Web 管理端）。
- **安装包与自动更新：** electron-builder 目标——macOS `.dmg`/`.zip`（已签名 + 公证）、Windows `.exe`（NSIS）、Linux `.AppImage`/`.deb`/`.rpm`。自动更新走 electron-updater；边车运行时**随应用一起更新**（它是被打包的资源）——单一更新通道、用户只看到单一版本号。`lraVersion`/`protocolVersion` 协商意味着更新中途的版本错配会优雅降级而非崩掉。**代码签名**对公证（macOS）与 SmartScreen（Windows）都必需，**捆绑的 Bun 可执行也必须签名**（它作为子进程被拉起，未签名会触发 Gatekeeper）。
- **状态住在 `~/.plexus`（共享、唯一真相源）。** 桌面应用、命令行、服务端 TUI 读同一个家目录，用户装了桌面应用也能用命令行打同一个网关，无需迁移。Electron 自己的 app data（窗口状态、偏好）住在 Electron 的 `userData`，与运行时状态分开。
- **无头服务器分发（并行产物）：** 同一份 `bun build --compile` 运行时二进制独立发（无 Electron），服务器/TUI 场景用 `plexus serve`（无头）+ `plexus tui`（客户端）。一份运行时，两个分发外壳。

## 三、产品 / 交互设计

**贯穿整套体验的中心命题：透明即产品（Transparency is the product）。** Plexus 把你机器上的能力打包给智能体（AI-native：它会自动生成所需的技能 + 工作流），然后治理**谁**能用**什么**、**为什么**、**用多久**——并让你看到**谁做了什么**。重设计存在的理由，正是因为当前管理端的*组织逻辑*把这条弧线藏在了六个并列 tab 后面。

### 3.0 一套词汇（冻结，每个界面只用这些词）

| 术语 | 面向用户的定义 |
|---|---|
| **智能体（Agent）** | 一个调用方身份（自报的 `agentId`，如 `claude-code`）。连接密钥才是信任边界；agent id 只用于划界、约束、审计。 |
| **能力（Capability）** | 一个来源暴露的、可授予的东西（它的**动词**：读 / 写 / 执行）。 |
| **来源（Source）** | 能力的出处（Obsidian 库、NAS 文件系统、cc-master……）。有**来路（provenance）**：第一方 / 受管 / 扩展。 |
| **范围（Scope）** | 能力 + 动词子集 + 可选**约束**（`路径在 Inbox/ 下`）。 |
| **授权（Grant）** | 站立的信任：某智能体可用某范围，直到其**信任窗口**结束。是你要撤销的对象。 |
| **信任窗口（Trust-window）** | 授权存活多久：一次 · 1 小时 · 1 天 · 7 天 · 直到我撤销 · 自定义。 |
| **令牌（Token）** | 授权的短时（15 分钟）自动刷新视图。管道，你永远不去管它。 |
| **任务包（Bundle）** | 一个**任务授权（Mode-2）**：N 个范围授权 + 附带上下文，命名、授予某一个智能体、一次性批准。 |
| **待办（Pending）** | 等你决策的请求（Mode-1 审批、任务包请求、或扩展注册）。 |
| **目的（Purpose）** | 智能体自报的自由文本「为什么是现在」——*「智能体说」*。渲染，但绝不信任。 |
| **叙述（Narration）** | 网关撰写的大白话解释——*「Plexus 说」*。是对抗注入、制衡「目的」的那一方。 |

两个支配信息架构、绝不能模糊的概念：
- **Mode 1 —— 临时审批。** 逐操作。「智能体 X，为了做 Z，想要 Y。批准一次 / 批一个窗口 / 拒绝。」
- **Mode 2 —— 任务包。** 预授权一个范围化任务包给某个智能体；它随后在范围内**静默**工作；越界则回落到 Mode 1。

### 3.1 五大界面，各司其职

Plexus 是**一个 Electron 应用**监督**一个无头运行时**（Bun 边车子进程）。应用是这一个运行时之上的五个界面，每个干一件不同的活，彼此交接。运行时是唯一真相源；每个界面都是它 HTTP/SSE 接口的一个视图 + 一个动作。

```
                         ┌──────────────────────────────────────────┐
                         │   Plexus 运行时（Bun 边车，无头）          │
                         │   授权 · 任务包 · 待办 · 审计 · 推送流     │
                         └───────────────▲──────────────────────────┘
            SSE 推送（待办/审计）          │   HTTP（读 / 动作）
        ┌──────────────┬─────────────────┼──────────────┬──────────────┐
        ▼              ▼                 ▼              ▼              ▼
    ┌────────┐   ┌───────────┐   ┌──────────────┐  ┌──────────┐  ┌──────────┐
    │  托盘   │   │   通知     │   │   管理界面    │  │  仪表盘   │  │   引导    │
    │ 状态    │──▶│ 审批       │──▶│  信任管理     │◀─│  概览     │  │  首次运行 │
    │ 一瞥    │   │（原生）    │   │（深度）       │  │（中枢）   │  │（弧线）   │
    └────────┘   └───────────┘   └──────────────┘  └──────────┘  └──────────┘
```

**一句话概括：** 托盘 = 常驻心跳 + 恐慌开关 · 通知 = Mode-1 来找你 · 管理界面 = 深度管理站立信任 · 仪表盘 = 此刻在发生什么 + 我信任了什么 · 引导 = 把弧线感受一次。

**托盘（Tray，常驻心跳）：** 一瞥状态 + 待审批数 + 那两三个无需开窗的动作。它是应用的永久存在；主窗口是用完即弃的。菜单栏图标状态：`◆` 平静（运行中、0 待办）· `◆²`（2 个待办角标）· `◇`（暂停/停止）· `⚠`（运行时错误）。

```
┌─ Plexus ─────────────────────────────┐
│ ● 运行中 · 3 个智能体活跃              │   ← 状态行（绿/琥珀/灰点）
│ 2 个审批等待中             ⌥⌘P  ▸     │   ← 仅 >0 时显示；打开审阅窗口
│──────────────────────────────────────│
│ 打开仪表盘                     ⌘D     │
│ 打开管理界面…                  ⌘,     │
│──────────────────────────────────────│
│ 最近                                  │
│   ✓ claude-code · 读 Index.md   2分前 │   ← 实时审计脉搏（最近 3 条），点击 → 活动
│   ✓ codex · 写 Inbox/note.md    9分前 │
│──────────────────────────────────────│
│ 暂停 Plexus  （拒绝一切）             │   ← 恐慌开关：停掉运行时
│ 退出                                  │
└──────────────────────────────────────┘
```

**通知（Notifications，Mode-1 的归宿）：** 把审批（或值得注意的事件）**带到**用户面前，让决策只需一瞥而非切换上下文。这就是 Mode-1 默认住的地方——管理端的「待办」tab 退化为*回退列表*，不再是主路径。三种原生通知原型：

**(a) Mode-1 审批** —— 核心卡片，可一瞥即决：

```
┌ Plexus ───────────────────────────────────┐
│ claude-code 想要写入你的 Obsidian 库       │   标题 = 谁 + 动词 + 什么
│ "把今天的会议记录归档到                     │   副标题 = 智能体的目的
│  Inbox/2026-06-24.md"                       │   （引号 = 智能体说）
│  ⚠ 扩展来源 · 写                            │   来路 + 敏感度标签
│ [ 批准一次 ] [ 批准 1 天 ] [ 拒绝 ]        │   ← 动作 = 信任窗口选择
│                              审阅… ↗       │   ← 打开完整审阅窗口
└────────────────────────────────────────────┘
```

两个批准按钮是网关对此「来路 × 动词」推荐的两个信任窗口（此处因为是扩展写，故为「一次」和「1 天」）。**审阅…** 打开窗口去选任意窗口 / 检查叙述 / 改投智能体。**拒绝**永远一击即达。关闭通知不等于批准。

**(b) 任务包请求** —— 智能体提议一个任务包：

```
┌ Plexus ───────────────────────────────────┐
│ claude-code 提议一个任务：                  │
│ "整理 NAS Inbox" —— 3 项能力                │
│ [ 审阅任务包… ]              [ 拒绝 ]      │   ← 任务包永不一击批准，必开审阅
└────────────────────────────────────────────┘
```

任务包授予更广的站立信任 → **绝不在通知里一击批准**，它打开审阅窗口的任务包视图。

**(c) 值得注意的事件**（可配置，默认仅撤销确认开启）：「已把 `fs.write` 授予 codex 7 天 · 撤销」「一个授权已过期」「来源 `obsidian` 离线」。这些是*信息性*的、可关闭的，至多带一个「撤销」。

**管理界面（Admin，深度）：** 你*去*的地方（相对于通知*来*找你），管理站立信任、来源、智能体，读审计。是这次重设计的重心——但日常已不再是 Mode-1 的住处。

**仪表盘（Dashboard，中枢）：** 回答「Plexus 此刻在做什么、我信任了什么？」一屏、以读为主、主窗口的默认落地界面。处处下钻进管理界面。

**引导（Onboarding，弧线，感受一次）：** 安装 → 「这是什么」→ 连接一个智能体 → 添加一个来源 → 见证一次真实调用 + 它的授权 + 它的审计。把心智模型弧线*感受*出来，然后溶解进仪表盘。

### 3.2 重设计的管理界面信息架构（IA）

**为什么当前六个 tab 失败：** 今天是 `能力 · 来源 · 待办 · 授权 · 令牌 · 审计` 六个**并列** tab，没有脊柱。心智模型是一条*弧线*（这是什么 → 它打包能力 → 按智能体授权 → 审计），但 tab 是一串*工程师实现出来的名词*的扁平清单。「令牌」是被告知永远不用管的管道，却是顶层 tab。「能力」（你从中授予的账本）和「授权」（站立账本）是两个 tab 在做一件事的两半。「待办」让 Mode-1 住在 tab 里，它本该住在通知里。完全没有「智能体」视图——尽管*按智能体*的信任才是全部要点。

**新脊柱：围绕心智模型弧线组织，而非围绕名词。** IA 是一个**左侧边栏**（桌面，不是 Web tab），其顺序*就是*那条弧线，分三段：

```
┌─────────────────────┬───────────────────────────────────────────────────┐
│  PLEXUS    ◆ 运行中  │   < 选中的小节渲染在这里 >                        │
│  ───────────────────│                                                   │
│  ▸ 概览        [中枢]│                                                   │
│                     │                                                   │
│  我暴露了什么        │   ← 第 1 段："它打包了我机器的能力"               │
│  ▸ 来源             │                                                   │
│  ▸ 能力             │                                                   │
│                     │                                                   │
│  我信任谁            │   ← 第 2 段："按智能体、按场景的授权"             │
│  ▸ 智能体           │      （新脊柱）                                   │
│  ▸ 审批         (2) │      （Mode-1 历史 + 待办回退）                   │
│  ▸ 任务授权         │      （Mode-2 任务包）                            │
│  ▸ 站立授权         │                                                   │
│                     │                                                   │
│  发生了什么          │   ← 第 3 段："审计 —— 谁做了什么，一清二楚"       │
│  ▸ 活动             │                                                   │
│  ───────────────────│                                                   │
│  连接密钥 ⧉          │   ← 常驻页脚（引导一个智能体接入）               │
│  设置 · 帮助         │                                                   │
└─────────────────────┴───────────────────────────────────────────────────┘
```

**为什么这条三段式弧线干净地修好了「组织逻辑不清晰」：**

- **概览** = 中枢（回答「这是什么 / 在发生什么」）。
- **「我暴露了什么」段** = *供给*侧：来源（能力出处）→ 能力（可授予面 + 它们生成的 AI-native 技能/工作流）。
- **「我信任谁」段** = *需求/授权*侧，两个模式的心脏：**智能体**（新脊柱——信任按调用方来看）、**审批**（Mode-1 记录 + 回退批准）、**任务授权**（Mode-2 任务包）、**站立授权**（扁平账本 + 撤销）。
- **「发生了什么」段** = **活动**（审计）。

关键洞见：**这三段本身就在教心智模型**——IA 仅凭存在就把模型教会了；而信任是**按智能体**被看见的。模型的实质是「给**不同智能体**、在不同场景、不同的能力集，各自独立授权」，把信任**按智能体**呈现，就是把心智模型渲染出来。站立授权（扁平）和任务授权（任务包）是同一份数据，按能力/任务包重新切分，给习惯那样想的时候用。

**每个旧功能都有新家，无一丢失：**

| 旧 tab / 功能 | 新家 | 变化 |
|---|---|---|
| 能力账本（从中授予） | **能力** | 变成*目录/检视器*，不再是主授予面。授予现在从智能体、来源或审批发起。显示附带技能 + 传递性工作流授权。 |
| 「授予访问」发起条 | **智能体 ▸ [agent] ▸ 授予…** 和 **能力 ▸ [cap] ▸ 授予给…** | 同一个组合器，从*谁*和*什么*两侧都可达。 |
| 来源（增/探测/启/停/删） | **来源** | 功能不变；新增引导探测卡片 + 行内密钥录入。 |
| 添加 Obsidian / 密钥 | **来源 ▸ 添加来源** | 同；改述为「连接一个来源」。 |
| cc-master 安装块 | **来源 ▸ 添加来源 ▸ 第一方适配器** | 折进「添加来源」，不再是松散的标题块。 |
| 待办（Mode-1 + 注册） | **通知（主）** + **审批（回退列表 + 历史）** | Mode-1 移到原生通知；审批保留列表（应对错过通知）+ 永久记录。 |
| 待办任务包请求 | **审批 ▸ 任务包请求** + 审阅窗口 | 分组卡片保留。 |
| 待办扩展注册 | **审批 ▸ 注册** | 安全敏感的 cli-bins/rest-hosts/跨源面**逐字保留**。 |
| 授权（站立账本 + 撤销） | **站立授权** | 扁平的每（智能体，能力）账本。也在**智能体**下分组呈现。 |
| 新任务授权组合器 | **任务授权 ▸ 新任务授权** | 为桌面重新构想。 |
| 任务包分组授权视图 | **任务授权** | 每任务包 = 一张卡；撤销任务包。也在智能体下。 |
| **令牌 tab** | **降级** → 智能体 ▸ [agent] ▸「此刻活跃」+ 活动 | 不再顶层。作为实时「活跃会话」细节展示，永远不作为「要去管理的东西」。设置 ▸ 高级里有开关给高级用户看原始令牌列表。 |
| 审计 | **活动** | 改用用户的词；新增过滤（按智能体/能力/结果）+ 实时脉搏。 |
| 连接密钥块 | **侧边栏页脚 + 智能体 ▸ 连接** | 常驻，外加「连接一个智能体」的第一步。 |
| 网关/标题状态 | **侧边栏头部标签 + 概览** | `● 运行中 · vN · 协议` 那一行。 |

#### 关键屏的线框（节选）

**智能体（新脊柱；按调用方看信任）：**

```
智能体
─────────────────────────────────────────────────────────────────────
 claude-code            此刻活跃 · 2 令牌     8 授权 · 1 任务包    ▸
 codex                  空闲                  3 授权               ▸
 plexus-cli             空闲                  1 授权               ▸
─────────────────────────────────────────────────────────────────────
 [ + 连接一个智能体 ]   ← 粘贴连接密钥 或 引导式集成

 ── claude-code ──────────────────────────────────────────── (展开) ─
   此刻活跃：  正在读 obsidian.vault.read （令牌 12 分钟后过期）
   站立授权 (8)：
     obsidian.vault.read   [读]   7天 · 5天后结束        [撤销]
     obsidian.vault.write  [写]   1天 · 18小时后结束      [撤销]
   任务授权 (1)：
     "整理 NAS Inbox"  3 能力 · 1天   [审阅] [撤销任务包]
   最近活动：  ✓ 读 Index.md 2分前 · ✓ 写 note 9分前  → 活动
   [ 授予一项能力… ]   [ 新任务授权… ]   [ 撤销全部 ]
```

**审批（Mode-1 记录 + 错过通知时的回退）：** 「审阅…」行打开与通知相同的审阅窗口；注册卡片保留完整安全面。若通知关闭或错过，Mode-1 可在此完整完成——无一丢失。

### 3.3 核心交互：审阅窗口（四段式）

Mode-1 走「通知 → 托盘 → 审阅窗口」。审阅窗口是现有 `PendingCard` 的四段式版式（智能体说 → Plexus 说 → 范围 → 控制），被抬进一个聚焦的桌面窗口。**对抗注入的铁律成立**：「智能体说」永远不与「Plexus 说」相邻或合并。

```
┌ 审阅审批 ─────────────────────────────────────────────────────┐
│  claude-code   ·   客户端：Claude Code 1.4   ·  ⚠ 扩展        │  谁 + 客户端标签
│  ───────────────────────────────────────────────────────────  │
│  智能体说                                                     │  (1) 智能体说 ——
│  "把今天的会议记录归档到 Inbox/2026-06-24.md"                  │      引号、隔离
│  ───────────────────────────────────────────────────────────  │      （对抗注入）
│  PLEXUS 说                                                    │  (2) 网关叙述
│  这会授予 obsidian.vault 的「写」—— 智能体能在你的库里创建和   │
│  修改笔记。扩展来源：Plexus 总会先问你。敏感度：中。          │
│  ───────────────────────────────────────────────────────────  │
│  范围   obsidian.vault.write  [写]                            │  (3) 范围/约束
│         ↳ （无约束 —— 整个库）                                │
│  智能体请求：1 天  （仅供参考 —— 你来决定）                   │
│  ───────────────────────────────────────────────────────────  │
│  授予给智能体： [ claude-code ▾ ]                              │  (4) 控制
│  信任窗口：    [ 一次 · 1时 · ◉1天 · 7天 · 直到我撤销 · …]    │
│           [ 批准 ]                          [ 拒绝 ]          │
└──────────────────────────────────────────────────────────────┘
```

### 3.4 核心交互：任务授权组合器（Mode-2）

两个入口：**所有者自己组合**（任务授权 ▸ 新任务授权）或**智能体提议**（通知 → 审阅任务包）。同一个形状。

```
┌ 新任务授权 ──────────────────────────────────────────────────┐
│ 任务名：    [ 整理 NAS Inbox                      ]          │
│ 授予给：    [ claude-code ▾ ]      窗口：[ 1 天 ▾ ]          │
│ ───────────────────────────────────────────────────────────  │
│ 范围内的能力                                                 │
│  [ nas.fs.read   ▾ ] [读]              （无约束）       ✕   │
│  [ nas.fs.move   ▾ ] [写] 仅 [ 路径在 ] Inbox/ 下       ✕   │
│  [ nas.fs.delete ▾ ] [写] 仅 [ 路径在 ] Inbox/Trash/ 下 ✕   │
│  [ + 添加能力 ]                                              │
│ ───────────────────────────────────────────────────────────  │
│ 给智能体的上下文                                             │
│  附带技能：  [ inbox-conventions ✕ ] [ + ]                  │
│  任务说明（会变成一个技能）：                                │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 把每条 Inbox 捕获按日期移进 Inbox/YYYY/MM/。            │ │
│  │ 绝不碰 Inbox/ 之外的任何东西。                          │ │
│  └────────────────────────────────────────────────────────┘ │
│ ───────────────────────────────────────────────────────────  │
│  预览：  3 个授权 · 限定在 Inbox/ 内 · 1 天后过期           │
│                         [ 创建任务授权 ]   [ 取消 ]          │
└──────────────────────────────────────────────────────────────┘
```

创建后任务包出现在「任务授权」和「智能体 ▸ claude-code」下。智能体在范围内**静默工作**——范围内调用不发通知。**confinement（限定）**：任何*越界*调用都回落到 Mode-1，弹通知，与 4a 一模一样。这条回退正是让宽泛预授权可以被接受的安全栏。（所有者经受信管理面组合的任务包直接创建；智能体*提议*的任务包必须 PEND 待批——这是「反自授权」的命门。）

### 3.5 仪表盘（概览中枢）

默认落地界面。以读为主；每块都下钻进管理界面。回答两个问题：*此刻在发生什么*，*我信任了什么*。


### 3.7 组件策略：复用 + 重组 + 换肤，不重写

现有组件编码了来之不易的正确性：四段式审批卡（智能体说 vs Plexus 说的对抗注入）、信任窗口选择器 +「来路 × 动词」默认、带约束的任务包组合器、分组任务包视图、审计分组。把这些保留为共享组件库；重设计 = 新的 IA 外壳 + 新界面（托盘/通知/仪表盘/引导）+ 一次换肤，高价值卡片（审阅窗口、组合器）抬进聚焦的桌面窗口。数据层（`api.ts`）与协议契约不变。

## 四、11 条关键决策（供拍板）

| # | 决策 | 一句话理由 |
|---|---|---|
| 1 | **嵌入模型：Bun 边车，受 Electron 主进程监督**（不移植进 Node）。 | 分离保持真实、运行时原封不动、崩溃隔离干净；移植会溶解分离并绑死发布节奏。 |
| 2 | **Web 管理端：保留，降级为可选的运行时托管回退**（同一份 React 代码，按新 IA 换肤）；桌面端为规范体验。 | 一份 UI 代码、两个外壳；对无头 Linux 服务器近乎免费；不分叉 UI。 |
| 3 | **跨平台：与桌面工作并行，先 Linux 再 Windows。** | 接缝已隔离、不阻塞 Electron；Linux 更简单且是服务器/TUI 目标，Windows 有 spawn/PATHEXT 细节。 |
| 4 | **框架：选 Electron 而非 Tauri。** | 托盘/通知/自动更新生态的成熟度（正是北极星点名的三个特性）压过包体积；团队 JS/TS 技能与 React 管理端直接对接。 |
| 5 | **Monorepo：`packages/{protocol,runtime,cli,desktop,web-admin,tui}`，`protocol` 是基石。** | 共享 `protocol` 包让契约由编译器跨运行时与所有客户端强制。 |
| 6 | **推送传输：SSE（`GET /v1/events`），不用 WebSocket。** | 服务端→客户端单向流正好是托盘/通知/审计脉搏的需求；智能体平面已证明 SSE 穿过回环护栏无需升级握手。 |
| 7 | **密钥交接：Electron 主进程读 `~/.plexus/connection-key` → `contextBridge` → 渲染进程**（免粘贴）。 | 复用已验证的管理密钥护栏、零新增加密、密钥不进页面全局；身份姿态不变。 |
| 8 | **IA：三段式侧边栏弧线，以 Agents 为主线；令牌降级；Mode-1 → 通知。** | 三段本身就在教心智模型，信任按调用方看，正好修好「组织逻辑不清晰」。 |
| 9 | **引导：有引导的脊柱，每步可跳过。** | 回报（步骤 4 一次真实调用 + 授权 + 审计）让模型「咔哒」一下点通；但绝不把应用挡在引导之后。 |
| 10 | **通知：审批*请求*必通知（阻塞），信息性事件默认仅高风险（设置里可调）。** | Mode-1 请求必须被决策；Mode-2 范围内调用静默是其价值所在，越界回退才通知。 |
| 11 | **组件：复用 + 重组 + 换肤，不重写。** | 现有卡片编码了对抗注入、信任窗口默认、约束等来之不易的正确性，契合「形式化 + 扩展 + 换肤」的总基调。 |

## 五、分阶段路线图（每阶段都可独立演示/交付）

> 工作量估计：S ≈ 数天，M ≈ 1–2 周，L ≈ 3–5 周，可并行处见标注。

**P0 —— Monorepo 拆包 + protocol 包 + 边车接缝。**（S，低风险）
从 `src/protocol/types.ts` 抽出 `packages/protocol`；建工作区；把运行时/命令行/Web 管理端搬进包，**行为零变化**；运行时获得受监督的 `plexus serve` 入口（就绪行 + 端口文件 + 监听适配器接缝）。
*演示：* 同一个产品、更干净的边界；运行时可作为受监督进程运行（CI 用现有测试证明对等）。

**P1 —— LRA v1 + 管理事件流。**（S–M）
加上轻端点 `GET /v1/{status,health,config,connection-key/rotate}`；`/admin/api/*` 在 `/v1` 下做别名；新增 `pending_added`/`pending_resolved`/`audit_appended` 事件 + `GET /v1/events` 管理 SSE。
*演示：* Web 管理端丢掉轮询，获得实时的待办/审计推送。

**P2 —— Electron 外壳（macOS）：监督 + 托盘 + 原生通知 + 恐慌开关。**（L）
Electron 应用在渲染进程里**先装进当前的管理端**；监督方（拉起/健康/重启/单实例）；连接密钥交接；托盘；由事件流驱动的原生 Mode-1 审批通知。
*演示：* 某智能体的 `plexus call` 弹出一条原生通知 → 从通知里批准 → 调用完成，**不开任何窗口**。

**P3 —— 管理界面 IA 重设计。**（L，可与 P2 重叠）
三段式侧边栏；Agents 主线；所有旧 tab 重映射到新屏；审阅窗口 + 任务授权组合器窗口；在稳定的 LRA 之上给共享组件换肤。
*演示：* 管理界面沿心智模型弧线导航；无一功能丢失。

**P4 —— 引导。**（M）
4 步首次运行弧线，预先解释 TCC 时刻。
*演示：* 全新机器 → 安装 → 4 步 → 用户已做出一次真实的、被审计的、被授权的调用。

**P5 —— 仪表盘。**（M）
概览中枢（此刻活跃 / 需要你 / 实时活动脉搏 / 按智能体的站立信任 / 暴露健康）。成为默认落地界面。
*演示：* 一屏回答「Plexus 在做什么 + 我信任了什么」，处处可下钻。

**P6 —— 跨平台 + Web/TUI + 打包分发。**（各 M，可从 P2 起并行）
填 Linux→Windows 平台接缝；发布换肤后的 Web 管理端回退；搭起 Linux 服务器 TUI；electron-builder 打包（签名/公证、自动更新、捆绑运行时可执行）。也在此折入延后的鉴权发现限定（按会话的可见性过滤），作为重设计后客户端/运行时的一部分。

**到「日常可用的 macOS 桌面 app」的关键路径 = P0 → P1 → P2**（之后 P3 把它变成重设计后的体验）。Windows 桌面需加 P6 的 Win32 接缝；跨平台运行时 + TUI 可并行。

## 六、结尾：版本姿态与设计不变量

**版本姿态：** 下一个版本 **≥ 0.5.1、去掉 rc、不急于发布。** 这是一次扎实的形式化 + 扩展，不是赶发布的重写；版本协商（`lraVersion`/`protocolVersion`）让运行时/客户端的版本错配优雅降级，给我们从容推进的空间。

**设计不变量（贯穿每个阶段）：**

- **透明即产品** —— 永远不为了「显得顺滑」而藏起授权/为什么/窗口/审计。
- **对抗注入** —— 「智能体说」（目的）永远与「Plexus 说」（叙述）视觉隔离；智能体的话绝不为网关的话上色或与之合并。
- **反自授权** —— 智能体的*请求*（Mode-1）或*提议的任务包*永远 PEND 给人类；只有受信的本地管理面才直接授予。
- **默认拒绝** —— 能力未授予即「未点亮」；UI 处处声明这一点。
- **一套词汇** —— 只用第 3.0 节的词，在每个界面、每条通知里。
- **连接密钥才是信任边界** —— 在任何展示按智能体信任的地方都说清楚；轮换 = 撤销全部。
